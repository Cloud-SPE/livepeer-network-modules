# Payment wire compatibility

The non-negotiable invariant on the `Livepeer-Payment` HTTP header is:

> **`livepeer.payments.v1.Payment` bytes MUST be byte-identical to
> `net.Payment` bytes produced by go-livepeer.**

A receiver written against go-livepeer's `pm/` package must be able to
decode a payment produced by this network's sender, and vice versa. Any
deviation is a wire break and must be caught at build time, not at
runtime in the field.

This document pins the contract.

## Why bytes-identical and not "schema-equivalent"

Payment envelopes ride the unmodified Livepeer probabilistic-micropayment
protocol. The on-chain TicketBroker contract expects ticket hashes
computed over a specific protobuf encoding; redeeming a winning ticket
fails on-chain if the receiver re-encoded the wire bytes during transit.
Schemas can drift while still validating; bytes cannot. We pin bytes.

## What is in scope

Five messages, re-declared in `proto/livepeer/payments/v1/types.proto`
with field numbers and wire types identical to go-livepeer's
`net/lp_rpc.proto`:

| Message | go-livepeer source | Carries |
|---|---|---|
| `PriceInfo` | `net/lp_rpc.proto` lines 102–116 | `price_per_unit`, `pixels_per_unit`, `capability` (uint32), `constraint` (string) |
| `TicketParams` | `net/lp_rpc.proto` lines 443–468 | `recipient`, `face_value`, `win_prob`, `recipient_rand_hash`, `seed`, `expiration_block`, `expiration_params` |
| `TicketSenderParams` | `net/lp_rpc.proto` lines 471–479 | `sender_nonce`, `sig` |
| `TicketExpirationParams` | `net/lp_rpc.proto` lines 482–488 | `creation_round`, `creation_round_block_hash` |
| `Payment` | `net/lp_rpc.proto` lines 494–509 | `ticket_params`, `sender`, `expiration_params`, repeated `ticket_sender_params`, `expected_price` |

Field names may differ between the two projects (snake_case here per
`buf` lint conventions; mixed case in go-livepeer). **Protobuf encoding
keys on field number, not name** — bytes on the wire are identical. The
package name also differs (`net` vs `livepeer.payments.v1`), which does
not affect binary encoding.

## What is out of scope

These exist in our spec but are NOT covered by the byte-compat contract,
because they never enter the gateway↔orchestrator wire path:

- `PayerDaemon` and `PayeeDaemon` gRPC service definitions — these are
  daemon↔consumer-app contracts inside a single host. Free to evolve.
- `OfferingPrice`, `CapabilityEntry`, `GetQuoteRequest`,
  `GetQuoteResponse`, `OpenSessionRequest`, `OpenSessionResponse`,
  `DebitBalanceRequest`, etc. These are RPC envelopes around the wire
  `Payment`, not the wire `Payment` itself.
- The capability-name ↔ capability-ID mapping. Consumer apps send
  string capability names to the daemon; the daemon maps them to the
  uint32 stored in `PriceInfo.capability` at the wire boundary.
- Generated language bindings, filenames, import paths.

## What we do NOT redeclare

Anything else in `lp_rpc.proto` (`OrchestratorInfo`, `SegData`,
transcoding RPCs, AI RPCs) is out of scope. Those belong to the
gateway↔orchestrator discovery and transcoding protocols that live
outside the payment surface.

## Signature wire-compat

The `sig` bytes inside `TicketSenderParams` carry a 65-byte ECDSA
signature in the format `[R || S || V]`, where `V ∈ {27, 28}`. The
signature is over `accounts.TextHash(ticket.Hash().Bytes())` — i.e., the
ticket hash wrapped in the Ethereum `personal_sign` EIP-191 prefix:

```
\x19Ethereum Signed Message:\n32
```

This format is non-negotiable. Any signer that omits the prefix or
serializes `V` outside `{27, 28}` will produce signatures that
go-livepeer-compatible receivers cannot recover the sender address from,
and on-chain redemption will revert.

Reference implementation: go-livepeer's `eth/accountmanager.go::Sign +
signHash` and `pm/ticket.go::Hash`. The prior reference impl at
`livepeer-cloud-spe/livepeer-modules-project/payment-daemon/internal/pm/`
extracts both verbatim.

## How compat is enforced

### 1. Field-level equivalence (static)

`buf breaking` runs in CI against `proto/livepeer/payments/v1/types.proto`
with the v1 schema as the lock. Any field-number change, wire-type
change, or wire-shape rename breaks the build. The v1 schema is pinned
at the first tagged release of this monorepo; post-release the only
allowed changes inside `v1/` are doc-comment edits.

> v0.2 status: `buf breaking` integration is open work. In the meantime,
> reviewers eyeball the field numbers when editing `types.proto`. The
> tradeoff is acceptable pre-1.0.0 because there are no external
> consumers of v1 yet.

### 2. Byte-for-byte round-trip (dynamic)

A canonical `Payment` fixture lives at
`payment-daemon/internal/compat/testdata/payment-canonical.bin`. It is
generated by a small Go tool that imports
`github.com/livepeer/go-livepeer/net` and `pm`, populates a fully-loaded
`Payment` (multi-ticket batch, non-trivial `PriceInfo`, realistic
`TicketParams`), and writes the marshaled bytes.

The test then:

1. Reads `payment-canonical.bin`.
2. Decodes the bytes with `livepeer.payments.v1.Payment`.
3. Re-encodes with our Go code.
4. Asserts the output bytes match the input bytes.
5. Decodes again and asserts field values match the original struct's
   Go representation.

A fail at any step is a wire break.

> v0.2 status: the test scaffold ships under
> `payment-daemon/internal/compat/`. The canonical fixture is a zero-byte
> placeholder until plan 0016 brings in the go-livepeer dep that
> generates it. The test currently passes vacuously on an empty fixture;
> plan 0016 makes it load-bearing.

### 3. Periodic re-generation against `go-livepeer/main`

A doc-gardening CI job re-runs the fixture generator against the latest
`go-livepeer` main branch and compares the resulting bytes to the
committed fixture. A mismatch is a flag, not an auto-update — the job
alerts; humans investigate. A mismatch usually means go-livepeer has
itself broken its wire format, which is worth flagging upstream.

> v0.2 status: deferred to plan 0016 alongside the canonical fixture.

## What we do NOT guarantee

- **Deterministic encoding in every runtime.** Protobuf does not guarantee
  map ordering. Both sides use the same Go encoder so this is fine in
  practice; spec-wise we promise "decode-identical" more strongly than
  "encode-identical".
- **Backwards-incompatible proto-lite variants.** Some environments use
  proto-lite encoders that skip unknown fields differently. We test the
  standard encoder only.

## Adding a new RPC without breaking v1

v1 message definitions and RPC signatures are frozen at first tag.
Functionality additions go in a new package directory:

```
proto/livepeer/payments/v1beta2/
```

Import v1 types where reused; declare new types only in v1beta2. v1beta2
graduates to v2 once stable. v1 remains in the repo indefinitely for
backwards compatibility. Same pattern Google uses for Kubernetes proto
evolution.
