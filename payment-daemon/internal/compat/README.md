# wire-compat round-trip

This directory enforces byte-identical wire compatibility with
go-livepeer's `net.Payment` proto type. The contract: any `Payment` that
go-livepeer emits must round-trip cleanly through our generated bindings
(`livepeer-network-protocol/proto-go/livepeer/payments/v1`) and produce
the same bytes when re-marshalled.

## Layout

- `wire_test.go` — the round-trip test (lives in the daemon's main
  module). Embeds `testdata/payment-canonical.bin` and asserts
  byte-identity after unmarshal-then-marshal.
- `testdata/payment-canonical.bin` — committed fixture, ~492 bytes,
  produced by `fixturegen/`.
- `fixturegen/` — separate Go module that imports
  `github.com/livepeer/go-livepeer/net` and writes the fixture. Lives in
  its own `go.mod` so go-livepeer's heavy transitive deps (libp2p,
  ipfs-core, ffmpeg cgo bindings) do not leak into the daemon binary's
  dependency graph.

## Regenerating the fixture

```
cd fixturegen
go run .
```

The committed fixture is checked in. Regenerate when:

1. The go-livepeer pin in `fixturegen/go.mod` is bumped (deliberate
   protocol-tracking action).
2. Our generated bindings change in a way that affects on-the-wire
   bytes (this should fail loudly via the round-trip test before the
   fixture is updated, surfacing the disagreement for human review).

The fixture is intentionally NOT regenerated by `go generate` or `make
proto`. A CI-side nightly drift check that re-pins go-livepeer @
`main`, regenerates, and diffs against the committed fixture is the
canary that surfaces upstream wire-format drift.

## Q3 pin

Per plan 0016 §11.Q3, the fixturegen `go.mod` pins
`github.com/livepeer/go-livepeer` at the latest stable v0.8.x tag at
plan-0016 implementation time (currently **v0.8.10**). Bumping requires
regenerating the fixture and ensuring the round-trip stays green.
